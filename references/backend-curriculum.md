# バックエンド・API 設計・システムデザイン 学習ガイド

> 北米テック企業のフルスタック / バックエンド面接で、サーバーサイドの仕組みや分散システムの設計を問われたときに自分の言葉で説明できるようになるためのカリキュラム。全セクション手を動かす演習形式。

---

## 1. サーバーサイドの基礎と実行の仕組み

### 1-1. Node.js のイベント駆動モデル
- `node:http` のみで GET / POST / PUT / DELETE を受け付ける簡易サーバーを書き、リクエストボディのパースを手動で `data` / `end` イベントで実装する
- `setTimeout` / `setImmediate` / `process.nextTick` の実行順序を混在させたコードを書き、Node.js のイベントループフェーズ（timers → pending → idle → poll → check → close）を体感する
- `fs.readFile`（コールバック）→ `fs.promises.readFile`（Promise）→ `async/await` の 3 パターンで同じファイル読み込みを書き、非同期処理の進化を比較する
- ブロッキング処理（巨大 JSON のパース、CPU 重い計算）をメインスレッドに置いたときのスループット低下を `autocannon` で計測し、Worker Threads に移して改善を確認する

### 1-2. HTTP プロトコルの深い理解
- `curl -v` で実際のリクエスト/レスポンスヘッダーを観察し、ステータスコード（200 / 201 / 204 / 301 / 302 / 400 / 401 / 403 / 404 / 409 / 429 / 500 / 503）の使い分けを一覧表にまとめる
- `Content-Type` / `Accept` / `Authorization` / `Cache-Control` / `ETag` / `If-None-Match` の各ヘッダーの役割を、curl で実際にリクエストして動作を確認する
- HTTP/1.1 の Keep-Alive と HTTP/2 の多重化の違いを図に描き、Chrome DevTools の Protocol カラムで実際の接続を確認する
- Cookie の属性（`HttpOnly` / `Secure` / `SameSite` / `Path` / `Domain` / `Max-Age`）をそれぞれ変えて設定し、ブラウザの DevTools Application タブで挙動を確認する

### 1-3. プロセスとコンテナの基礎
- `docker build` / `docker run` で Node.js アプリをコンテナ化し、ホストとコンテナのポートマッピングを実験する
- マルチステージビルドで本番イメージのサイズを最小化する Dockerfile を書き、ステージありなしのイメージサイズを比較する
- `docker-compose` で Node.js + PostgreSQL + Redis の 3 コンテナ構成を立ち上げ、サービス間通信を実験する
- 環境変数（`process.env`）でデータベース接続文字列やシークレットを注入する仕組みを実装し、ハードコードとの違いを体感する

### 1-4. ログ・モニタリング・可観測性
- `console.log` の代わりに構造化ログ（JSON 形式 + タイムスタンプ + リクエスト ID + ログレベル）を出力するロガーを自作する
- ログレベル（DEBUG / INFO / WARN / ERROR）のフィルタリングを環境変数で切り替える仕組みを実装する
- リクエストごとに一意の ID を割り当て、その ID をログ・レスポンスヘッダーに含めるミドルウェアを作る（分散トレーシングの基礎）
- `process.on('uncaughtException')` / `process.on('unhandledRejection')` でプロセスが落ちる前にログを書く仕組みを実装する

## 2. データベース設計とクエリ最適化

### 2-1. リレーショナルデータベースの基礎（PostgreSQL）
- EC サイトのスキーマ（users / products / orders / order_items / reviews）を正規化して設計し、ER 図を手書きする
- 各テーブルの `CREATE TABLE` 文を書き、主キー・外部キー・NOT NULL・UNIQUE・CHECK 制約を適切に設定する
- 1NF → 2NF → 3NF の正規化プロセスを、非正規化テーブルから段階的に分解する演習を行う
- 意図的に非正規化が有利なケース（読み取りが圧倒的に多いアクセスパターン）を考え、正規化版と非正規化版のクエリパフォーマンスを比較する

### 2-2. SQL クエリの実践
- `JOIN`（INNER / LEFT / RIGHT / FULL OUTER / CROSS）を全パターン書き、結果の違いをデータで確認する
- サブクエリ vs JOIN vs CTE（`WITH` 句）で同じ結果を出すクエリを 3 通り書き、`EXPLAIN ANALYZE` で実行計画を比較する
- Window 関数（`ROW_NUMBER` / `RANK` / `DENSE_RANK` / `LAG` / `LEAD` / `SUM OVER`）で「各カテゴリ内の売上ランキング」「前月比売上」などを計算する
- トランザクション（`BEGIN` / `COMMIT` / `ROLLBACK`）を使い、送金処理（A の残高を減らして B の残高を増やす）のアトミック性を実験する
- 分離レベル（READ UNCOMMITTED / READ COMMITTED / REPEATABLE READ / SERIALIZABLE）を変えて、ダーティリード・ファントムリードの発生を確認する

### 2-3. インデックスとクエリ最適化
- `EXPLAIN ANALYZE` で Seq Scan と Index Scan の違いを確認し、実行時間を比較する
- B-Tree インデックス / GIN インデックス（全文検索）/ GiST インデックス（地理データ）の使い分けを、それぞれのユースケースで実験する
- 複合インデックス（`CREATE INDEX ON orders(user_id, created_at)`）で、カラムの順序が検索パフォーマンスに与える影響を実験する
- インデックスが効かないパターン（関数適用 / 暗黙の型変換 / `LIKE '%keyword'`）を実験し、回避策を実装する
- スロークエリログを有効にし、遅いクエリを特定 → インデックス追加 → 改善の一連の流れを実践する

### 2-4. NoSQL データベース（MongoDB / Redis）
- MongoDB で同じ EC サイトのデータをドキュメント設計し、RDB の設計と比較する（埋め込み vs 参照の判断基準を整理する）
- MongoDB の `aggregate` パイプライン（`$match` / `$group` / `$lookup` / `$unwind` / `$project`）で売上集計レポートを作る
- Redis の基本データ型（String / Hash / List / Set / Sorted Set）を全て使ったコードを書き、それぞれのユースケースを整理する
- Redis でセッションストア、レート制限カウンター、リアルタイムランキングをそれぞれ実装する
- RDB + Redis のキャッシュパターン（Cache Aside / Read Through / Write Through / Write Behind）を実装して動作を比較する

### 2-5. データマイグレーションと運用
- マイグレーションツール（knex / prisma migrate / golang-migrate いずれか）でスキーマ変更（カラム追加・リネーム・削除）を管理する
- ゼロダウンタイムマイグレーションの手順（新カラム追加 → 書き込みを両方に → 読み取りを切り替え → 旧カラム削除）を実際に手を動かして実装する
- シードデータの投入スクリプトを書き、テスト環境のセットアップを自動化する

## 3. API 設計の原則と実装

### 3-1. RESTful API 設計
- REST のリソース命名規則（名詞・複数形・ネスト）を使って EC サイトの API エンドポイント一覧を設計する（`GET /products`, `POST /orders`, `GET /users/:id/orders` など）
- HATEOAS（レスポンスに次のアクション用のリンクを含める）を実装し、クライアントがリンクを辿って操作を進められるようにする
- ページネーション 3 パターン（offset / cursor / keyset）を実装し、大量データでのパフォーマンスを比較する
- フィルタリング（`?category=food&price_min=500`）、ソート（`?sort=-created_at`）、フィールド選択（`?fields=id,name,price`）を実装する
- API バージョニング（URL パス `/v1/` / ヘッダー `Accept: application/vnd.api.v2+json` / クエリ `?version=2`）の 3 方式を実装して比較する

### 3-2. GraphQL の基礎と実装
- スキーマ定義（`type Query`, `type Mutation`, `type Subscription`）を書いて、同じ EC サイトの API を GraphQL で設計する
- リゾルバを実装し、N+1 問題が発生する状況を `console.log` で確認する
- DataLoader を導入して N+1 問題を解決し、クエリ数の変化を比較する
- REST で 3 回のリクエストが必要だったデータ取得を、GraphQL の 1 クエリで実現し、レスポンスサイズとリクエスト数を比較する
- Subscription でリアルタイム通知（新しい注文が入ったら画面を更新）を実装する

### 3-3. 認証・認可
- セッションベース認証を実装する（ログイン → セッション ID 発行 → Cookie にセット → サーバーでセッション検証）
- JWT 認証を実装する（ログイン → アクセストークン + リフレッシュトークン発行 → Authorization ヘッダーで送信 → 検証ミドルウェア）
- JWT のペイロードを `base64` でデコードして中身を確認し、署名が改ざんを防いでいることを実験する
- セッション認証と JWT 認証のメリット・デメリットを表にまとめる（スケーラビリティ / ステートレス / 無効化の容易さ / セキュリティ）
- OAuth 2.0 の Authorization Code Flow を図に描き、Google OAuth で実際にログインを実装する
- RBAC（Role-Based Access Control）のミドルウェアを作る（admin / user / guest ロールで API アクセスを制限する）

### 3-4. 入力バリデーションとエラーハンドリング
- リクエストボディのバリデーション（必須チェック / 型チェック / 文字列長 / 正規表現 / カスタムルール）をミドルウェアで一元化する
- エラーレスポンスのフォーマットを統一する（`{ error: { code, message, details } }`）
- HTTP ステータスコードとエラーコードの対応を設計し、ドキュメント化する
- SQL インジェクション / XSS / CSRF の攻撃を実際に仕掛けて、バリデーションとサニタイズで防ぐ実験をする
- レート制限を Token Bucket アルゴリズムで実装し、`429 Too Many Requests` を返す仕組みを作る

### 3-5. API ドキュメントとテスト
- OpenAPI（Swagger）仕様書を YAML で書き、Swagger UI で確認する
- API テストを 3 層で書く（ユニットテスト：ビジネスロジック単体 / インテグレーションテスト：DB 含む / E2E テスト：HTTP リクエスト）
- テスト用のデータベースを Docker で立ち上げ、テスト前にシードデータ投入 → テスト後にクリーンアップする仕組みを作る
- Postman のコレクションを作り、API の動作を手動で素早く検証できるようにする

## 4. バックエンドアーキテクチャパターン

### 4-1. レイヤードアーキテクチャ
- Controller → Service → Repository の 3 層に分けて EC サイトの API を構築する
- 各レイヤーの責務を明確にする（Controller：HTTP の入出力変換、Service：ビジネスロジック、Repository：データアクセス）
- Repository を抽象化して、PostgreSQL から MongoDB に差し替えてもService 層が変更不要であることを実証する
- 依存性注入（DI）を手動で実装し、テスト時にモックリポジトリを注入してサービス層を単体テストする

### 4-2. メッセージキューと非同期処理
- Redis（Bull/BullMQ）または RabbitMQ でメッセージキューを構築する
- 同期的に処理していた重い処理（メール送信・画像リサイズ・PDF 生成）をキューに移し、API のレスポンス時間が改善することを計測する
- ワーカープロセスを複数起動し、並列処理でスループットが向上することを確認する
- 失敗したジョブのリトライ戦略（指数バックオフ / 最大リトライ回数 / Dead Letter Queue）を実装する
- イベント駆動アーキテクチャの簡易版を実装する（注文確定イベント → 在庫引当 + メール送信 + 分析データ送信をそれぞれ別のワーカーで処理）

### 4-3. WebSocket とリアルタイム通信
- `ws` ライブラリで WebSocket サーバーを構築し、ブラウザのコンソールから `new WebSocket()` で接続する
- チャットルーム機能を実装する（接続管理 / ルーム参加・退出 / ブロードキャスト / ダイレクトメッセージ）
- Socket.IO を使って同じ機能を書き直し、自動再接続・フォールバック（Long Polling）の仕組みを確認する
- WebSocket 接続のスケーリング問題を理解する（ステートフル接続 → Sticky Session / Redis Pub/Sub による複数サーバー間の同期）

### 4-4. CQRS とイベントソーシング（概念理解）
- 読み取り用モデルと書き込み用モデルを分離した API を実装する（注文の書き込みは正規化 DB、注文履歴の読み取りは非正規化ビュー）
- イベントストア（イベントを追記のみで保存するテーブル）を作り、イベントのリプレイで現在の状態を再構築する仕組みを実装する
- CQRS / イベントソーシングが適切なケースと過剰なケースを整理し、判断基準を言語化する

## 5. システムデザイン

### 5-1. システムデザイン面接のフレームワーク
- 以下の 4 ステップを全ての問題に適用する練習をする
  1. **要件の明確化**（5 分）：機能要件と非機能要件を質問で引き出す（DAU / QPS / レイテンシ要件 / データ量 / 一貫性要件）
  2. **概算**（5 分）：QPS / ストレージ / 帯域幅をざっくり計算する（back-of-the-envelope estimation）
  3. **ハイレベル設計**（15 分）：主要コンポーネントのブロック図を描く
  4. **深掘り**（15 分）：面接官が気になるポイントを詳細に設計する

### 5-2. Back-of-the-Envelope Estimation 練習
- 以下の数値を暗記して素早く概算できるようにする
  - 1 日 = 86,400 秒 ≈ 10 万秒
  - 1 Million requests/day ≈ 12 QPS
  - 1 Billion requests/day ≈ 12,000 QPS
  - 1 KB × 1 Million = 1 GB
  - 1 MB × 1 Million = 1 TB
- 「Twitter の 1 日の投稿量からストレージを概算する」「YouTube の動画配信に必要な帯域幅を概算する」などの練習問題を 10 問解く

### 5-3. スケーラビリティの基本コンポーネント
- 以下の各コンポーネントの役割を図に描いて説明できるようにする
  - **ロードバランサー**：L4 vs L7、ラウンドロビン / Weighted / Least Connections / IP Hash のアルゴリズムを比較する
  - **CDN**：Pull 型 vs Push 型、キャッシュ無効化の仕組みを図で整理する
  - **リバースプロキシ**：Nginx で静的ファイル配信 + API プロキシの設定を実際に書く
  - **水平スケーリング vs 垂直スケーリング**：メリット・デメリットを表にまとめる
  - **データベースレプリケーション**：Master-Slave 構成を Docker Compose で構築し、書き込みは Master、読み取りは Slave に振り分ける実験をする
  - **データベースシャーディング**：ハッシュベース / レンジベース / ディレクトリベースの戦略を比較し、リシャーディングの課題を整理する

### 5-4. キャッシュ戦略の深掘り
- キャッシュの 5 つのパターンを実装して比較する
  - Cache Aside（Lazy Loading）
  - Read Through
  - Write Through
  - Write Behind（Write Back）
  - Refresh Ahead
- キャッシュの一貫性問題を実験する（DB を更新したのにキャッシュが古いまま → TTL / イベントベース無効化 / Cache Aside の更新順序）
- キャッシュ雪崩（Cache Stampede）を再現し、TTL のジッター / ロック / 事前ウォーミングで対策する実験をする
- Memcached vs Redis のアーキテクチャの違い（マルチスレッド vs シングルスレッド、データ構造のサポート）を整理する

### 5-5. 一貫性とトレードオフ
- CAP 定理を具体例で理解する（ネットワーク分断時に一貫性を取るか可用性を取るかのシナリオを 3 パターン考える）
- 強い一貫性 / 最終的一貫性 / 因果的一貫性の違いを、チャットアプリのシナリオで説明する
- 分散トランザクションの 2 パターンを理解する
  - Two-Phase Commit（2PC）：コーディネーター障害時の問題を図で整理する
  - Saga パターン：オーケストレーション型とコレオグラフィー型を比較する
- 冪等性（Idempotency）の重要性を理解し、冪等キーを使った API を実装する（同じリクエストを 2 回送っても結果が変わらないことを確認する）

### 5-6. 定番システムデザイン問題の演習

各問題で「要件整理 → 概算 → ハイレベル設計 → 深掘り」の流れを 45 分以内に完成させる練習をする。

#### URL 短縮サービス（TinyURL）
- ハッシュ生成（MD5 / Base62 / カウンターベース）の選択肢を比較する
- 301 vs 302 リダイレクトの使い分けを整理する
- 読み取り重視のキャッシュ戦略を設計する
- クリック分析（Analytics）のデータパイプラインを追加設計する

#### チャットアプリ（Slack / WhatsApp）
- 1:1 チャットとグループチャットのメッセージ配信モデルを設計する
- オンライン/オフラインステータスの管理方法を設計する
- メッセージの永続化と既読管理のスキーマを設計する
- WebSocket のスケーリング（Redis Pub/Sub でサーバー間同期）を設計する

#### ニュースフィード（Twitter / Instagram）
- Fan-out-on-write vs Fan-out-on-read のトレードオフを整理する
- 有名人（フォロワー数百万人）のツイートの特殊処理を設計する
- フィードのランキングアルゴリズム（時系列 vs エンゲージメント重み付け）を概念設計する
- メディア（画像・動画）の配信パイプラインを設計する

#### 動画配信サービス（YouTube / Netflix）
- 動画のアップロード → トランスコード → 配信のパイプラインを設計する
- Adaptive Bitrate Streaming（ABR）の仕組みを図で整理する
- CDN の配置戦略と動画のチャンク配信を設計する
- コメント・いいね・視聴回数のカウントシステムを設計する

#### ECサイト（Amazon）
- 商品カタログ / 検索 / カート / 注文 / 決済のマイクロサービス分割を設計する
- 在庫管理の一貫性問題（同時注文で在庫がマイナスになる問題）を解決する
- 検索サービス（Elasticsearch）の導入と同期の仕組みを設計する
- 注文 → 決済 → 配送の Saga パターンを設計する

#### レート制限サービス（API Gateway）
- Token Bucket / Leaky Bucket / Fixed Window / Sliding Window の 4 アルゴリズムを図で比較する
- 分散環境でのレート制限（Redis ベースの共有カウンター）を設計する
- ルールエンジン（IP 単位 / ユーザー単位 / API エンドポイント単位）を設計する

#### 通知サービス（Push / Email / SMS）
- 通知テンプレートと配信チャネルの抽象化を設計する
- 配信保証（at-least-once）とデデュプリケーションの仕組みを設計する
- ユーザーの配信設定（オプトイン / オプトアウト / 配信頻度制限）を設計する
- 大量配信時のバックプレッシャーとキューイングを設計する

## 6. 総合演習：スケーラブルな API を段階的に構築する

### 6-1. モノリスで構築する
- Express + PostgreSQL + Redis でEC サイトの API を構築する（商品 CRUD / ユーザー認証 / カート / 注文）
- レイヤードアーキテクチャ（Controller / Service / Repository）で整理する
- JWT 認証 + RBAC ミドルウェアを実装する
- API テスト（Jest + Supertest）を全エンドポイントに書く

### 6-2. パフォーマンス問題を発見・改善する
- `autocannon` でベンチマークを取り、ボトルネック（遅い SQL / N+1 問題 / キャッシュミス）を特定する
- スロークエリにインデックスを追加し、Before / After を計測する
- Redis キャッシュを追加し、DB へのクエリ数が減ることを確認する
- コネクションプーリングを導入し、接続のオーバーヘッドを削減する

### 6-3. スケールアウトに対応する
- Docker Compose で API サーバーを 3 台に増やし、Nginx でロードバランシングする
- セッションを Redis に移し、どのサーバーにリクエストが来ても認証が通ることを確認する
- PostgreSQL の Master-Slave レプリケーションを構築し、読み取りを Slave に振り分ける
- 重い処理（メール送信・画像リサイズ）をメッセージキューに移し、ワーカーで非同期処理する

### 6-4. マイクロサービスに分割する
- モノリスの API を 3 つのサービス（商品サービス / 注文サービス / ユーザーサービス）に分割する
- サービス間通信を REST → gRPC に切り替え、レイテンシとペイロードサイズの変化を計測する
- API Gateway パターンを実装し、クライアントからは単一のエンドポイントに見えるようにする
- 分散トレーシング（OpenTelemetry）を導入し、リクエストがサービスをまたいで流れる様子を可視化する
- 6-1 のモノリスと比較して「コード量」「デプロイの独立性」「運用の複雑さ」を記録する
