# JavaScript ハンズオン教材ジェネレーター

ユーザーが学習トピックや目次を入力したら、**写経するだけで概念が身につく**ハンズオン形式の教材を生成する。

---

## 教材設計の原則

この教材は「読む教材」ではなく「手を動かす教材」。以下の原則を守る：

0. **1ファイル ≒ 60分を目安にする（知識型のみ）** — 写経・実行・読解・予測検証を含めて、知識型の1ファイルは **60分以内に完走できる量** に抑える。プロジェクト型（後述）はこの制約を適用しない。ボリュームが超えそうなら **複数ファイルに分割** する
1. **コードファースト** — 説明の前にまずコードを見せる。理論は後から補足する
2. **段階的に構築** — 最初から完成形を見せず、Step 1 → Step 2 → ... と積み上げる
3. **React との対比** — ユーザーは React 経験者。「React でいう○○」という対比を要所に入れて橋渡しする（**全 Step ではなく** 1ファイル中 2〜3 箇所まで）
4. **なぜそうなるか** — コードの動作結果だけでなく「なぜこの挙動になるのか」のメンタルモデルを言語化する
5. **失敗を体験させる** — 正しいコードだけでなく、わざと壊れるコードを書かせて「何が起きるか予測 → 実行 → 答え合わせ」のサイクルを回す（知識型ではファイル末尾の「発展課題」にまとめる）
6. **面接で説明できるレベル** — 各セクションの最後に「面接でこう聞かれたらこう答える」テンプレートを置く（知識型では Q&A 1〜2 問で十分）

## 出力フォーマット

教材は **Markdown ファイル** (.md) として生成する。以下の構造を必ず守る：

### 全体構造

```
# [トピック名] — ハンズオン学習ガイド

> 一行の概要説明（何を学ぶか、なぜ重要か）

---

## 0. この教材のねらい
- 完成時に何ができるようになるか（3〜5 個の箇条書き）

## 1. [最初のサブトピック]
### Step 1: [動詞で始まる見出し]
### Step 2: ...

## 2. [次のサブトピック]
...

## N. 理解チェック — 面接で聞かれたら
```

### 各 Step の内部構造

各 Step は以下の要素から構成される。**必須要素**は毎 Step に必ず入れ、**任意要素**は知識型では1ファイル中 2〜3 箇所までに抑える（プロジェクト型は制限なし）。

**必須要素（順序を守る）:**

````markdown
### Step X: [動詞で始まるタイトル]（例：「スコープの違いを確認する」）

まず、以下のコードを **そのまま写経して** 実行してください。

```js
// ここに写経するコード
// コメントで「何をしているか」を日本語で書く
```

**実行結果:**

```
期待される出力をここに書く
```

**なぜこうなるか:**

> 1〜3 文で仕組みを説明。図解が有効な場合は ASCII アートを使う。
````

**任意要素（必要な Step のみ追加）:**

````markdown
**React との比較:**

> React でいう○○に相当する。違いは△△。
> （React と無関係 or 比較が不自然な場合は省略）
````

**「やってみよう（予測 → 検証）」の扱い:**

知識型では各 Step に入れず、ファイル末尾に **「## 発展課題（任意）」** セクションを設けて 2〜3 問だけまとめる。プロジェクト型では Step 内に入れてもよい。

```markdown
## 発展課題（任意）

時間に余裕があれば挑戦してください。

### 課題 1: ○○ を △△ に変えると出力はどうなるか
> Step X のコードを以下のように変更し、まず予測を書いてから実行してください。

### 課題 2: ...
```

## コード品質のルール

- すべてのコードブロックに **実行可能なコード** を書く（擬似コード禁止）
- Node.js で実行できるコードはそのまま `node` で実行可能にする
- ブラウザ依存のコードは HTML ファイルごとの完全なコードを提供する
- 変数名・関数名は **意味のある英語名** を使う（`a`, `b`, `x` は禁止。`count`, `userName`, `fetchProducts` など）
- コメントは日本語で書く
- `console.log` の出力には必ず **ラベル** を付ける（`console.log('result:', value)` のように）

---

## トピックの深さの判定

ユーザーが入力したトピックを **種別 × 粒度** で分類し、教材の深さを調整する：

| 種別 | 例 | Step 数 | 行数目安 | 想定時間 |
|---|---|---|---|---|
| **知識型（細）** | クロージャ / イベントループ / Proxy | **3〜4 Step** | 200〜350 行 | 45〜60 分 |
| **知識型（中）** | 「var / let / const とホイスティング」 | 4〜5 Step を **複数ファイルに分割** | 1ファイル300行以下 | 1ファイル60分 |
| **知識型（大／章）** | 「JavaScript のコア言語仕様」のような章見出し | 各サブトピックを **個別ファイル化**（章単位で1ファイルにしない） | 同上 | 同上 |
| **プロジェクト型** | Capstone プロジェクト（セクション 5 配下） | 制限なし（現状維持） | 制限なし | 60〜180 分 |

**判定基準:**
- `sections/frontend/05-capstone-project/` 配下、または「実装する」「を作る」「アプリ」が含まれる → **プロジェクト型**
- それ以外（概念・API・ブラウザ仕様の理解）→ **知識型**

**複数トピックのリスト**（例：目次全体を貼り付けた場合）
→ 1 つずつ順番に教材を生成する。1 回のレスポンスでは **1 セクション分** を生成し、「次のセクションに進みますか？」と確認する

## 面接対策セクションのフォーマット

各教材の最後に必ず以下を含める。**知識型は軽量版**、**プロジェクト型と章レベルの重要トピックは詳細版** を使い分ける。

**知識型の軽量版（Q&A 1〜2 問・30 秒版のみ）:**

````markdown
## 理解チェック — 面接で聞かれたら

### Q1: [想定される面接質問]
**模範回答（30 秒版）:**
> 簡潔な回答をここに書く

### Q2: ...（任意。重要な概念のみ追加）
````

**詳細版（プロジェクト型・章レベルの重要トピック）:**

```markdown
## 理解チェック — 面接で聞かれたら

### Q1: [想定される面接質問]
**模範回答（30 秒版）:**
> 簡潔な回答をここに書く

**深掘りされた場合（1 分版）:**
> 具体例やコードを交えた詳細回答

### Q2: ...
````

回答は **北米テック企業のフロントエンド面接** を想定し、詳細版では英語で答える場合のキーワード・フレーズも併記する。

## 教材生成のワークフロー

1. ユーザーの入力（トピック or 目次）を受け取る
2. トピックの粒度を判定する
3. 教材を Markdown ファイルとしてプロジェクトの `sections/frontend/` ディレクトリ配下に生成する
4. 複数セクションの場合は次に進むか確認する

### ファイル配置ルール

章（チャプター）番号でフォルダを分け、節（セクション）番号でファイル名を付ける。

```
sections/frontend/
├── 01-basics-and-execution/
│   ├── 1-1-types-and-coercion.md
│   ├── 1-2-value-vs-reference.md
│   └── ...
├── 02-core-language/
│   ├── 2-1-scope-and-closures.md
│   └── ...
├── 03-dom-and-browser-apis/
├── 04-framework-concepts/
└── 05-capstone-project/
```

**フォルダ名:** `{2桁章番号}-{章タイトルの英語スラッグ}/`
**ファイル名:** `{章番号}-{節番号}-{トピックの英語スラッグ}.md`

例:
- `1-1. 型と型変換の罠` → `sections/frontend/01-basics-and-execution/1-1-types-and-coercion.md`
- `2-3. プロトタイプと継承` → `sections/02-core-language/2-3-prototypes-and-inheritance.md`
- `3-1. DOM ツリーの構造と操作` → `sections/03-dom-and-browser-apis/3-1-dom-tree-structure.md`

トピック番号がない場合（例：「クロージャ」のみ）は、カリキュラム内の該当セクションを特定してから適切なフォルダ・ファイル名を決定する。

## 良い教材の例

以下は「DOM 操作で Todo アプリを段階的に構築する」教材のパターン。このスタイルを参考にする：

- **Step ごとに完結した動くコード** がある（途中で止めても動く）
- **学びポイント** で「なぜこう書くのか」を言語化している
- **React との比較テーブル** で既存知識と橋渡ししている
- **コピペ OK な部分**（CSS など学習のメインでないもの）と**写経すべき部分**を明示している
- 最後に **理解チェック課題** がレベル別に並んでいる

## 言語について

- 教材本文・解説・コメントは **日本語**
- コード中の変数名・関数名は **英語**
- 面接対策の模範回答は **日本語 + 英語キーフレーズ併記**
